Request Flow

  1. Frontend sends POST /orders with JWT Bearer token
  2. Order Service persists order + outbox event in a single transaction
  3. Background scheduler polls outbox every 1s, publishes to Kafka
  4. Notification Service consumes events from Kafka
  5. Notification Service pushes real-time updates via WebSocket

2. Design Decisions

Messaging Technology: Apache Kafka

AlternativeWhy Kafka was chosen instead
RabbitMQKafka provides higher throughput. Partitioning enables parallel consumption. Disk retention enables replay after crashes.
Direct HTTPTight coupling. If notification service is down, order creation fails.
Redis Pub/SubNo persistence. Messages lost if consumer is down. No consumer groups.

Communication Pattern: Outbox Pattern

Solves the dual-write problem — order and event written in the same DB transaction. Background scheduler publishes to Kafka independently.

Direct publish (PROBLEMATIC):
  1. Save to DB     ✓
  2. Publish Kafka   ✗ fails → data inconsistency

Outbox Pattern (WHAT WE USE):
  1. Save order to DB     }  SAME TRANSACTION
  2. Save outbox event    }  Both succeed or both rollback
  3. Scheduler publishes to Kafka independently
  → Always consistent. Eventual delivery guaranteed.

Database: PostgreSQL

RequirementHow PostgreSQL meets it
ACID transactionsOutbox Pattern needs order + event in same transaction
Unique constraintsidempotency_key UNIQUE prevents duplicates under race conditions
Performance10,000+ TPS. With HikariCP (20 conn × 5ms) = ~4,000 inserts/sec capacity

Scaling Strategy

StrategyImplementationImpact
Stateless servicesJWT auth, no server sessionsHorizontal scaling behind load balancer

Security: JWT Authentication

  • JWT on all /orders/** endpoints
  • Rate limiting on /auth/login (5 attempts/min per IP)
  • CORS restricted to http://localhost:3000
  • Actuator metrics require JWT; health endpoints public
  • Input validation with structured error responses
  • Custom 401 responses (not Spring's default 403)

3. Failure Handling

3.1 Kafka Unavailable

Order is still created successfully. Outbox Pattern decouples order creation from Kafka:

1. POST /orders → order + outbox event saved in SAME transaction → 201 Created
2. Scheduler tries to publish → Kafka down → fails
3. Circuit breaker tracks failures:
   - <50% failure rate → retry next poll (1s)
   - ≥50% failure rate → circuit OPENS for 10s (fast-fail)
   - After 10s → HALF_OPEN → 3 test calls
4. Event stays PENDING in outbox
5. Kafka recovers → scheduler publishes all pending events

No data lost. Orders persisted. Events eventually delivered.

3.2 Service Crashes

ScenarioWhat happensRecovery
Order Service crash before commitTransaction rolls back, no partial stateClient retries with same idempotencyKey
Order Service crash after commitOrder saved, outbox event PENDINGRestart → scheduler resumes publishing
Notification Service crashKafka retains unacked messagesRestart → consumer resumes from last committed offset

Three layers of protection:

LayerMechanismHandles
1. ApplicationfindByIdempotencyKey() → return existing99% of duplicates
2. DatabaseUNIQUE constraint on idempotency_keyRace conditions
